## El contexto: un servidor MCP local que debería funcionar

Hace unos días audité e instalé `codebase-memory-mcp`, un servidor MCP local para persistir contexto de proyectos. El binario responde, el handshake es correcto y la interfaz web carga. Todo indica que debería estar operativo. Sin embargo, Devin Desktop no lo conecta.

La razón no está en el servidor. Está en una **política de organización impuesta por el backend de Devin** y en una cadena de documentación contradictoria que hace imposible saber dónde colocar la configuración correcta.

## Caso 1: la allowlist fantasma

`devin mcp list` muestra el servidor, pero con un estado inesperado:

```text
Status: Blocked — Server is not in your organization's allowlist
```

El detalle viene de `devin auth status`:

```text
MCP servers: enforced (allowlist: 0 entries, registries: 0)
```

Aquí hay dos problemas superpuestos:

- El plan es **Pro**, no Enterprise, así que no hay panel de admin para editar la allowlist.
- La documentación oficial dice que, si la allowlist está vacía, todos los servidores están permitidos. El CLI hace lo contrario: con `enforced` y 0 entradas, bloquea todo.

Probé con dos tokens de API: un `cog_*` de servicio y una personal key legacy. Ambos autentican en `/v3/self`, pero los endpoints de `team_settings` y `mcp_allowlist` devuelven 404 o 403. No hay API pública para arreglarlo desde un plan Pro.

La documentación de troubleshooting del CLI empeora la contradicción. En la sección [MCP server issues](https://docs.devin.ai/cli/troubleshooting#mcp-server-issues), el síntoma "Tools not appearing" remite a "Verify your enterprise allows MCP servers in Team Settings". La guía de MCP Configuration añade: "A server can be blocked if it isn't on your team's allowlist or in an enforced MCP registry". Ambas fuentes asumen un plan Enterprise con panel de administración; en Pro esa política no debería poder activarse, pero el backend la aplica igual.

**Reproducibilidad:** ejecutar `devin auth status` en cualquier org Pro con `MCP servers: enforced` y allowlist vacía. El resultado es bloqueo total de servidores locales.

**Impacto:** cualquier servidor MCP local —Stripe, Notion, Playwright, Vercel, o uno propio— queda inutilizable sin intervención de soporte.

## Caso 2: dónde vive realmente user_settings.pb

La documentación de Devin Desktop FAQ sitúa el archivo de settings del usuario en:

```text
~/.codeium/user_settings.pb
```

En el sistema real el archivo está en:

```text
~/.codeium/windsurf/user_settings.pb
```

Para comprobarlo, creé una copia del archivo en `~/.codeium/` y monitoricé ambos con `inotifywait`. Devin Desktop solo accedió al de `~/.codeium/windsurf/`. Incluso reiniciando el language server, la copia en la raíz de `.codeium/` no recibió ni un solo `OPEN`.

**Reproducibilidad:** copiar `user_settings.pb` a `~/.codeium/` y observar con `inotifywait -m -e open,access` mientras se reinicia el language server de Devin Desktop.

**Impacto:** si sigues la docs, colocas la config en el lugar equivocado y Devin Desktop no la lee. El fallo es silencioso: no hay error, simplemente no funciona.

## Caso 3: la ruta del MCP config también miente

La FAQ señala `~/.codeium/mcp_config.json` como el MCP config de Devin Desktop. La documentación del Devin CLI describe que, si se activa `read_config_from.windsurf`, importa la configuración de Windsurf desde el directorio canal-específico:

```text
~/.codeium/<channel>/mcp_config.json
```

Para la versión stable, eso es:

```text
~/.codeium/windsurf/mcp_config.json
```

Esas rutas son de Windsurf/Cascade, no de Devin CLI. Las ubicaciones correctas del CLI son:

| Nivel   | Ruta                          |
| ------- | ----------------------------- |
| User    | `~/.config/devin/config.json` |
| Local   | `.devin/config.local.json`    |
| Project | `.devin/config.json`          |

La configuración nativa de usuario (`~/.config/devin/config.json`) ahora está estructurada y alineada con el schema. En `permissions` se usan patrones `Exec(...)` para agrupar familias de comandos:

- `Exec(git)`, `Exec(gh)`, `Exec(npm run)`, `Exec(pnpm run)`, `Exec(docker compose)`, `Exec(uv run)` para agrupar permisos por herramienta.
- `mcp__*` en `allow` para permitir todos los MCP, tanto los del marketplace como los custom.
- `ask` y `deny` explícitos para operaciones sensibles (`Exec(rm -rf)`, `Exec(docker system prune)`, etc.).
- `env` con `${env:VAR}` para secrets como `notion` y `stripe`, delegando el valor a variables de entorno exportadas.

El CLI no documenta ningún fallback a `~/.codeium/mcp_config.json`. En la práctica, tener el servidor en `~/.codeium/windsurf/mcp_config.json` sin `read_config_from.windsurf` no lo hace visible para `devin mcp list`.

**Reproducibilidad:** activar `read_config_from.windsurf` en `~/.config/devin/config.json`, colocar el servidor solo en `~/.codeium/mcp_config.json` y ejecutar `devin mcp list`. No aparecerá.

**Impacto:** duplicidad de archivos de configuración y confusión sobre cuál es la fuente de verdad. El usuario termina editando el archivo incorrecto.

## Caso 4: `~/.config/Devin/` versus `~/.config/devin/`

La documentación de transición de Windsurf a Devin Desktop menciona `~/.config/Devin/` como ruta de configuración. En el sistema coexisten dos directorios con nombres casi iguales:

| Ruta               | Contenido real                              | Uso                            |
| ------------------ | ------------------------------------------- | ------------------------------ |
| `~/.config/Devin/` | Cache, IndexedDB, cookies, logs de Electron | Datos de la aplicación gráfica |
| `~/.config/devin/` | `config.json`, `AGENTS.md`, `skills/`       | Configuración del agente CLI   |

Para verificarlo, oculté `~/.config/devin/` y mantuve `~/.config/Devin/`. El CLI creó un nuevo `~/.config/devin/` vacío con un `config.json` mínimo. La aplicación, por su parte, recreó `~/.config/Devin/` para su caché.

**Reproducibilidad:** renombrar `~/.config/devin/` temporalmente, ejecutar `devin auth status` y observar que se crea un nuevo directorio con la misma ruta en minúscula.

**Impacto:** confusión entre configuración de agente y datos de aplicación. Si el usuario borra `~/.config/Devin/` pensando que es cache, pierde datos de la app; si borra `~/.config/devin/`, pierde reglas y configuración del agente.

## Caso 5: la sección `/settings/windsurf` promete API keys que no gestiona

En la interfaz web, la sección `/settings/windsurf` declara:

> "Devin Desktop — Manage your Devin Desktop API keys and privacy settings."

Las API keys reales se gestionan desde `/settings/devin-api`. El panel de Windsurf no tiene el control que anuncia en su propio título.

**Reproducibilidad:** navegar a `app.devin.ai/settings/windsurf` y luego a `/settings/devin-api`. Comparar los controles disponibles.

**Impacto:** el usuario busca las API keys donde la interfaz le dice, no las encuentra y pierde tiempo hasta descubrir la ruta correcta.

## Caso 6: el language server muere por detrás

Durante el diagnóstico, Devin Desktop empezó a emitir:

```text
[Error] windsurf client: couldn't create connection to server.
Error: connect ECONNREFUSED 127.0.0.1:46815
```

El puerto es efímero; el problema no es el puerto, sino que el proceso `devin-desktop` ha crasheado con `SIGTRAP`/`ud2` (un `DCHECK` intencional de Chromium). El language server queda huérfano y el cliente VS Code entra en bucle de reconexión.

El crash file está en `/var/crash/_opt_Devin_devin-desktop.1000.crash`.

**Reproducibilidad:** no encontré un paso exacto; parece un crash preexistente del binario de Devin Desktop en Ubuntu 26.04.

**Impacto:** la IDE se vuelve inusable hasta cerrar y reiniciar Devin Desktop.

## Caso 7: sintaxis de permisos inválida (`Bash(...)` en vez de `Exec(...)`)

La sección `permissions` del config usaba entradas como `Bash(git status:*)`. El Devin CLI no reconoce esa sintaxis: solo acepta `Exec(...)`. El resultado es que las reglas `Bash(...)` se leen como texto descriptivo y se ignoran silenciosamente.

Esto tiene un efecto peligroso. Si el usuario intenta restringir git con `Bash(git status:*)` en `ask` o `deny`, la restricción no se aplica. En cambio, cualquier entrada `Exec(git)` en `allow` auto-aprueba TODO git: `git push`, `git reset --hard`, `git branch -D`, etc. La intención de seguridad se invierte: los comandos que deberían pedir confirmación o estar bloqueados pasan sin control.

La corrección es usar `Exec(...)` por comando concreto:

```text
Exec(git status)
Exec(git push)
Exec(git log)
Exec(rm -rf)
```

El schema no acepta comodines ni el prefijo `Bash`. Cada patrón de permiso debe ser una acción exacta.

## Caso 8: MCP servers HTTP con `serverUrl` en vez de `url`

Los servidores MCP HTTP en Devin CLI esperan la clave `url`. En Windsurf/Devin Desktop se usa `serverUrl`. Tener `serverUrl` en el config del CLI es otro residuo de la configuración de Windsurf que el CLI no interpreta.

Los servidores afectados (`cloudflare`, `context7`, `deepwiki`) se migraron a `url`. Una vez corregido, el CLI los reconoce como endpoints HTTP válidos.

## Caso 9: secrets con `env` vacíos

Los servidores `notion` y `stripe` tenían sus campos `env` vacíos. El CLI no falla con un mensaje claro; simplemente no envía el token al servidor. La corrección es declarar la variable de entorno esperada:

```text
${env:NOTION_TOKEN}
${env:STRIPE_SECRET_KEY}
```

Esto requiere que el usuario exporte `NOTION_TOKEN` y `STRIPE_SECRET_KEY` en la sesión antes de ejecutar `devin`. El config ahora sigue el schema oficial: no contiene valores secretos, solo referencias a variables de entorno.

## Caso 10: el cache de team settings no contiene la política MCP

`devin auth status` informa que la política de org está en el servidor:

```text
MCP servers: enforced (allowlist: 0 entries, registries: 0)
```

El CLI guarda una copia caché de los team settings en:

```text
~/.cache/devin/cli/team_settings.bin
```

El archivo pesa 1019 bytes y se decodifica con `protoc --decode_raw`. El contenido estructural es:

| Campo | Tipo                 | Valor                    |
| ----- | -------------------- | ------------------------ |
| 1     | bool                 | true                     |
| 4     | bool                 | true                     |
| 12    | bool                 | true                     |
| 22    | bool                 | true                     |
| 23    | bool                 | true                     |
| 24    | bool                 | true                     |
| 7     | allowlist de modelos | 36 entradas              |
| 26    | string               | `claude-opus-4-7-medium` |
| 6     | int                  | dos IDs numéricos        |
| 30    | string               | `subagent-default`       |

Lo relevante es lo que **no** aparece:

- `allowlist` o `whitelist` de MCP.
- `mcp_servers`.
- `org_id`.
- `permissions`.
- `allowed_domains` / `denied_domains`.
- `enforced` como campo explícito.

Es decir, el caché local de team settings contiene la política de modelos, pero no contiene la política de bloqueo de MCP. Esa decisión llega de otra parte: server-side, otro endpoint, o un booleano sin etiquetar que no se expone en el archivo local.

**Reproducibilidad:** decodificar `~/.cache/devin/cli/team_settings.bin` con `protoc --decode_raw` y comparar con la salida de `devin auth status`.

**Impacto:** si intentas editar el caché local para desbloquear MCP, no encontrarás la palanca. La política no es modificable localmente.

## Caso 11: `.devin/config.local.json` y la trampa de gitignorear los secretos

El Devin CLI tiene tres niveles de configuración, pero solo dos de ellos están pensados para no mezclarse con el repo:

| Nivel   | Ruta                          | ¿Se sube a git?           |
| ------- | ----------------------------- | ------------------------- |
| User    | `~/.config/devin/config.json` | No, está en `$HOME`       |
| Local   | `.devin/config.local.json`    | No, debe estar gitignored |
| Project | `.devin/config.json`          | Sí, es parte del proyecto |

`.devin/config.local.json` es la ruta correcta para los overrides locales de un proyecto: permite definir `mcpServers` y `permissions` con prioridad sobre `~/.config/devin/config.json`, sin contaminar el archivo de proyecto que se versiona. En un caso real, ese archivo local incluyó permisos para patrones `mcp__*` y servidores específicos; `devin mcp list` los leyó correctamente y mostró `codebase-memory-mcp` como configurado. El bloqueo, sin embargo, siguió siendo el mismo: `MCP servers: enforced` con allowlist vacía.

Esto confirma dos cosas:

- La ruta local funciona y tiene precedencia sobre el config de usuario.
- Ningún override local puede saltar una política de organización aplicada por el backend.

Pero hay un riesgo de seguridad paralelo. Si el archivo local contiene tokens, API keys o rutas de servidores personales, y el directorio `.devin/` no está en `.gitignore`, es muy fácil commitear secretos sin darse cuenta. La práctica correcta es añadir `.devin/` al `.gitignore` global o al del proyecto. En este caso se creó un `.gitignore` a nivel de `$HOME` con esa línea para evitar que cualquier config local de Devin termine en un repositorio.

**Reproducibilidad:** colocar un servidor en `.devin/config.local.json`, ejecutar `devin mcp list` y ver que aparece; luego ejecutar `devin auth status` y confirmar que sigue bloqueado por org. Si no se gitignora `.devin/`, `git status` mostrará el archivo como untracked.

**Impacto:** el usuario puede tener la configuración local perfecta, funcionando y priorizada, pero sigue bloqueado por la org. Además, sin `.gitignore` adecuado, expone secretos en el control de versiones.

## Caso 12: `devin mcp list` incluye servidores que no están en `~/.config/devin/config.json`

Aunque `~/.config/devin/config.json` ahora solo contiene los servidores configurados explícitamente, `devin mcp list` sigue mostrando entradas adicionales: `filesystem`, `github`, `memory`, `proton-pass`. Ninguna de ellas aparece en el archivo local.

Lo más probable es que vengan de la configuración de la cuenta u organización, o que se hayan importado automáticamente desde Windsurf durante la transición. Esto confirma que `~/.config/devin/config.json` no es la única fuente de verdad: el backend de Devin mantiene un estado de MCP separado que se fusiona con la config local.

**Reproducibilidad:** comparar la salida de `devin mcp list` con el contenido de `~/.config/devin/config.json`. Los servidores extra no aparecen en el archivo.

**Impacto:** el usuario puede pensar que su config local está sucia o que hay duplicados, cuando en realidad la lista es una combinación de configuración local + estado de cuenta/org. Limpie el config.json local, pero los servidores heredados seguirán ahí.

## Qué he intentado y qué no funciona

- Configurar `codebase-memory-mcp` en `~/.config/devin/config.json` de forma nativa. La configuración anterior en `~/.codeium/mcp_config.json` y `~/.codeium/windsurf/mcp_config.json` era de Windsurf/Cascade y fue eliminada. El config de usuario ahora está limpio y alineado con el schema: permisos agrupados por `Exec(git)`, `Exec(gh)`, `Exec(npm run)`, `Exec(pnpm run)`, `Exec(docker compose)`, `Exec(uv run)`, patrón `mcp__*` para permitir todos los MCP, `ask`/`deny` explícitos para operaciones sensibles, y `env` con `${env:VAR}` para `notion` y `stripe`. También se verificó que `devin mcp list` sigue mostrando servidores heredados de la cuenta o de Windsurf (`filesystem`, `github`, `memory`, `proton-pass`), pero los servidores definidos localmente se leen correctamente. El bloqueo de org persiste.
- Reescribir la sección `permissions` de `Bash(...)` a `Exec(...)` con comandos concretos. La sintaxis `Bash(...)` era ignorada silenciosamente; `Exec(...)` es la única que el CLI aplica.
- Cambiar `serverUrl` a `url` en los servidores HTTP (`cloudflare`, `context7`, `deepwiki`). El CLI no reconoce `serverUrl`; es la clave de Windsurf/Devin Desktop.
- Reemplazar los `env` vacíos de `notion` y `stripe` por `${env:NOTION_TOKEN}` y `${env:STRIPE_SECRET_KEY}`, delegando los secretos a variables de entorno exportadas por el usuario.
- Probar tokens de API `cog_*` y `apk_*`. Autentican, pero no hay endpoint operativo para modificar la allowlist en plan Pro.
- Verificar la precedencia de `user_settings.pb` con `inotifywait`. El archivo real está en `~/.codeium/windsurf/`.
- Verificar el caso de `~/.config/devin/` y `~/.config/Devin/`. Son dos directorios con propósitos distintos.
- Decodificar `~/.cache/devin/cli/team_settings.bin` con `protoc --decode_raw`. El caché contiene políticas de modelos pero no la configuración de allowlist ni el flag de bloqueo de MCP.
- Colocar `mcpServers` y permisos en `.devin/config.local.json` y confirmar que `devin mcp list` lee el archivo. La lista muestra el servidor, pero el bloqueo de org sigue activo. También se añadió `.devin/` a `.gitignore` para evitar filtrar configuración local.

## Conclusión

La configuración local de `codebase-memory-mcp` ahora está limpia y alineada con el schema oficial. `~/.config/devin/config.json` usa permisos agrupados por `Exec(...)` (`git`, `gh`, `npm run`, `pnpm run`, `docker compose`, `uv run`), patrón `mcp__*` para permitir todos los MCP, `ask`/`deny` explícitos para operaciones sensibles, y `env` con `${env:VAR}` para secrets. `devin mcp list` reconoce los servidores configurados localmente, aunque sigue mostrando servidores heredados de la cuenta o de Windsurf (`filesystem`, `github`, `memory`, `proton-pass`). El servidor es funcional. El bloqueo de org persiste. El problema es una **intersección de doce factores**:

1. Una política de org (`MCP servers: enforced`) que no debería activarse en plan Pro sin panel de control.
2. Una documentación de troubleshooting que confirma la allowlist pero solo describe soluciones para Enterprise.
3. Rutas de settings (`user_settings.pb`) y MCP config que mezclan Devin Desktop, Windsurf y Devin CLI.
4. Dos directorios `~/.config/Devin/` y `~/.config/devin/` con propósitos distintos y nombres casi idénticos.
5. Una interfaz web que anuncia funcionalidad en una sección distinta de donde realmente está.
6. Un crash del proceso principal que deja el IDE sin language server.
7. Una sintaxis de permisos inválida (`Bash(...)`) que es ignorada y deja pasar comandos que deberían estar restringidos; la config actual usa `Exec(...)` correctamente.
8. Servidores HTTP configurados con `serverUrl` en lugar de `url`, lo que el CLI no reconoce; la config actual usa `url`.
9. Secrets con `env` vacíos que no fallan visiblemente, pero tampoco envían tokens al servidor; la config actual usa `${env:VAR}`.
10. Un caché de team settings (`team_settings.bin`) que descarga la política de modelos pero no la política de bloqueo de MCP, lo que confirma que el bloqueo no es local.
11. El uso correcto de `.devin/config.local.json` como override local, que funciona y tiene prioridad, pero no puede saltar una política de org; además, exige gitignorar `.devin/` para no commitear secretos.
12. Servidores MCP adicionales (`filesystem`, `github`, `memory`, `proton-pass`) que aparecen en `devin mcp list` pero no en `~/.config/devin/config.json`, lo que indica que el backend mantiene un estado de MCP separado de la config local.

La única salida documentada —y que no funciona en el plan Pro— es contactar a soporte de Devin. Hasta que la org se desbloquee o se actualice el plan, los servidores MCP locales seguirán bloqueados.

Si estás en la misma situación, verifica `devin auth status` antes de tocar la configuración local. Si ves `MCP servers: enforced` y `allowlist: 0 entries`, el problema no está en tu `~/.config/devin/config.json` ni en tu `.devin/config.local.json`; está en la org.
